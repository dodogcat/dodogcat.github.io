**기간**: 2024.09 - 2024.12  
**역할**: 온디바이스 추론 통합, 성능 계측, 병목 분석, 개선 방향 정리  
**기술 스택**: Unity, C#, ONNX, Barracuda, YOLOv3-tiny, YOLOv8, Xcode Instruments, iPad / 모바일 환경  
**코드**: 비공개 저장소, 요청 시 설명 가능

## 개요

모바일 환경에서 객체 인식 기능을 안정적으로 구동하기 위해 기존 **YOLOv3-tiny** 기반 구조를 **YOLOv8 계열**로 확장하고, Unity 내부에서 실제로 어떤 지연이 발생하는지 분석한 프로젝트입니다.

처음에는 최신 모델이 더 무거워서 화면이 끊긴다고 추정했지만, 실제 계측 결과 병목의 중심은 단순한 모델 크기보다 **후처리 단계와 Unity 코드 흐름**에 더 가까웠습니다. 이 프로젝트는 모델 교체 자체보다, 추론 파이프라인을 전처리, 추론, 후처리 단위로 나눠 병목을 분리해낸 경험에 더 큰 의미가 있었습니다.

## 문제

- **기존 인식 성능의 한계**  
  기존 구조에서는 원하는 수준의 인식 품질을 확보하기 어려웠고, 모델 변경과 파이프라인 재구성이 함께 필요했습니다.

- **YOLOv8 적용 후 체감 성능 저하**  
  YOLOv8n을 적용하자 화면이 끊기는 현상이 나타났습니다. 처음에는 모델이 무거워서 생긴 문제라고 판단했지만, 정량적인 계측 없이는 원인을 확정할 수 없었습니다.

## 수행 내용

- **모델 교체와 ONNX 통합**  
  YOLOv3-tiny에서 YOLOv8n으로 전환을 시도하고, Unity 환경에서 사용할 수 있도록 ONNX 기반 추론 파이프라인을 구성했습니다.

- **계측 구조 추가**  
  초당 프레임과 함께 전처리, 추론, 후처리 시간을 구분해서 볼 수 있는 계측 구조를 추가했습니다. 이 계측 정보를 통해 체감 성능 저하가 어느 단계에서 시작되는지 확인했습니다.

- **모델 크기별 비교**  
  YOLOv8n, YOLOv8s, YOLOv8m을 비교한 결과, 총 추론 시간은 각각 **67ms, 88ms, 147ms** 수준으로 차이가 있었지만 평균 프레임은 **40, 38, 38 FPS**로 큰 차이를 보이지 않았습니다. 이 결과로 모델 크기만으로 체감 지연을 설명하기 어렵다는 점을 확인했습니다.

- **단계별 병목 분석**  
  입력 크기와 후처리 비용의 관계도 별도로 확인했습니다.  
  예를 들어 **YOLOv8n(416)** 기준으로 전처리 4ms, 추론 4ms, 후처리 59ms가 측정됐고, **YOLOv8n(640)**에서는 후처리가 108ms까지 증가했습니다.  
  즉, 실제 병목은 모델 본체보다 **후처리 단계**에 더 가까웠습니다.

- **구조 변경 시도**  
  인식 성능을 높이기 위해 모델 구조 개량과 경량화도 함께 시도했습니다. 백본 경량화와 ONNX 양자화 옵션을 테스트했지만, 기대한 만큼의 유의미한 성능 이득은 크지 않았습니다.

- **런타임 호환성 검토**  
  YOLOv8은 Barracuda가 일부 Resize 옵션을 바로 지원하지 않아 설정을 조정해 사용했습니다. 반면 YOLOv10은 호환 문제가 남아 있었고, 이후 Sentis 적용 가능성까지 검토했습니다.

## 결과

- 모델 크기 증가 자체보다 **후처리와 Unity 코드 흐름**이 주요 병목이라는 점을 확인했습니다.
- 입력 해상도가 커질수록 후처리 비용이 급격히 증가한다는 점을 정량적으로 확인했습니다.
- Unity 기반 모바일 추론 파이프라인을 전처리, 추론, 후처리 단위로 나눠 반복 분석할 수 있는 구조를 만들었습니다.
- “더 가벼운 모델을 쓰면 해결된다”는 단순한 가설보다, 실제 코드와 파이프라인을 먼저 봐야 한다는 결론을 얻었습니다.

## 의미

이 프로젝트는 다음 역량과 연결됩니다.

- Unity 기반 온디바이스 추론 파이프라인 구축
- 계측 도구를 통한 병목 원인 분리
- 모델 성능과 실제 시스템 동작 사이의 차이 분석
- 런타임 호환성과 후처리 비용을 함께 고려한 최적화 판단
