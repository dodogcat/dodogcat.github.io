### 요약

Unity 기반 모바일 환경에서 객체 인식 기능을 개선하기 위해 기존 **YOLOv3-tiny** 구조를 분석하고, **YOLOv8 계열로의 전환 가능성**을 검토한 프로젝트입니다.  
실제 계측 결과, 체감 성능 저하의 주요 원인은 모델 자체보다 **후처리 단계와 Unity 코드 흐름**에 더 가까웠습니다.

새 모델 이식은 제안하고 검토했지만, 런타임 호환성 문제로 인해 실제 서비스 구조를 완전히 변경하지는 않았습니다. 대신 전처리, 추론, 후처리 시간을 분리해 병목을 확인하고, 어떤 조건에서 지연이 커지는지 정량적으로 분석했습니다.

### 문제

- 기존 구조에서는 원하는 수준의 인식 품질을 확보하기 어려웠습니다.
- YOLOv8n 적용을 검토하는 과정에서 화면 끊김이 발생했고, 원인이 모델 크기인지 파이프라인 구조인지 분리해서 확인할 필요가 있었습니다.

### 수행 내용

- YOLOv3-tiny 기반 모바일 추론 구조를 점검하고, YOLOv8 계열 적용 가능성을 검토했습니다.
- Unity 내부에 전처리, 추론, 후처리 시간을 나눠 볼 수 있는 계측 구조를 추가했습니다.
- YOLOv8n, YOLOv8s, YOLOv8m 비교를 통해 모델 크기만으로 체감 지연을 설명하기 어렵다는 점을 확인했습니다.
- 입력 크기와 후처리 비용의 관계를 분석한 결과, **YOLOv8n(416)**에서는 전처리 4ms, 추론 4ms, 후처리 59ms였고, **YOLOv8n(640)**에서는 후처리가 108ms까지 증가했습니다.
- Barracuda의 연산 호환성을 검토하며 실제 이식 가능 범위를 확인했습니다.

### 결과

- 주요 병목이 모델 본체보다 **후처리와 Unity 코드 흐름**에 있다는 점을 확인했습니다.
- 모바일 추론 파이프라인을 단계별로 분석할 수 있는 계측 구조를 만들었습니다.
- 새 모델 도입은 제안 단계까지 검토했으며, 실제 적용 여부는 런타임 호환성과 운영 안정성을 기준으로 보수적으로 판단했습니다.

### 의미

- 모바일 환경에서의 추론 파이프라인 분석
- 계측 기반 병목 원인 분리
- 모델 성능과 실제 시스템 동작 차이 해석
- 런타임 호환성을 고려한 기술 판단
